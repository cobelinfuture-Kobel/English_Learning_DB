# ULGA-S9Z2 Learner Event Log Design Scan

## 1. Scope

This design scan establishes the architectural boundary, taxonomy, and canonical JSON contract for the **Learner Event Log** (S9Z2) in the Universal Learning Graph Authority (ULGA) ecosystem. 

This task is **DESIGN SCAN ONLY**.
- No runtime code (Python files) is modified.
- No existing graph JSON files or validator files are modified.
- No learner state or candidate ranking code is modified.
- No mastery scoring algorithm or event reducers are implemented.
- No learner-specific real data is created.
- Names of real learners (e.g., James, Cyndi) are **not** hard-coded into the proposed schemas, example contracts, or machine-readable patterns.

---

## 2. Purpose

The ULGA project models educational mastery via a structured graph of learning targets (Grammar, Vocabulary, Chunks, Patterns, Themes) and dependency connections. Historically, downstream layers such as the **Learner State Builder** and **Candidate Ranking Engine** attempted to compute mastery state and recommendations directly from sparse updates or coarse heuristics.

### Why Learner Event Log is Needed

1. **Decoupling Event Capture from State Calculation**: An interface or system that directly mutates learner state upon receiving an event creates massive coupling. If the mastery formula, retention decay rate, or node dependency graph changes, the historic state cannot be corrected because the original raw context is lost.
2. **Idempotency and Replayability**: To support changes in pedagogical models or to recover from database corruption, the system must be able to rebuild the learner state from scratch. An immutable log of raw events is the only way to achieve deterministic state replay.
3. **Auditability and Debugging**: When the Antigravity Planner makes an unexpected recommendation, developers and instructors must be able to trace the decision back to the exact chronological list of interactions (e.g., hints used, response times, specific incorrect answers) that produced the underlying mastery score.

### Why Raw Event Evidence Must Precede State and Ranking

Without a canonical, immutable event log, the derived state represents a floating point of data without historical backing.
- **Learner State** cannot determine if a learner has "decayed" in a skill or if their performance was a lucky guess without knowing the exact distribution of practice attempts over time.
- **Candidate Ranking** cannot prioritize reinforcement versus introduction tasks accurately if it only knows the current mastery band (e.g., "practicing") but lacks the exposure history (e.g., whether the node was seen twice in a reading exercise or failed three times in a quiz).

## 3. Position in Roadmap

The Learner Event Log sits at the boundary between raw learner activity and the derived cognitive model of the learner. It acts as the chronological registry of all evidence.

```text
+-----------------------+
|  Downstream Systems   | (Reading / Dialogue / Exercise / Assessment Authorities)
+-----------+-----------+
            |
            | Emits raw interaction events
            v
+-----------+-----------+
|   Learner Event Log   | <--- [THIS DESIGN SCAN (S9Z2)]
+-----------+-----------+
            |
            | Processed by Reducers / Replay engines
            v
+-----------+-----------+
| Learner State Auth /  | <--- Ingests Exposure Mapping Bridge (S9Z1) & Learner Exposure Evidence (S9X/Y)
|   Mastery Graph       |
+-----------+-----------+
            |
            | Evaluated for next-step suitability
            v
+-----------+-----------+
|   Candidate Ranking   | (S10A)
+-----------+-----------+
            |
            | Selected and scheduled
            v
+-----------+-----------+
|  Antigravity Planner  | (S10D)
+-----------------------+
```

### Relationships with Roadmap Components

- **S9Z1 Exposure Mapping Bridge**: The Bridge maps static graph nodes to learning opportunities. The Learner Event Log captures when those opportunities are actually executed, using `learning_opportunity_id` and `target_nodes` as the matching keys.
- **Learner Exposure Evidence**: Generated by mapping the event log's `exposure_seen` and related events to opportunity overlap records, establishing a raw seen-count and confidence level.
- **Learner State Authority**: Learner State is the derived profile of the learner. The Learner Event Log is the source database that the State Authority aggregates (reduces) to compute scores, error rates, last-seen timestamps, and decay.
- **Mastery Graph**: Represents the dynamic overlay on top of the static ULGA graph, showing nodes as "unknown", "seen", "practicing", "functional", "mastered", or "automatic". It is derived entirely by playing the Learner Event Log forward.
- **S10A Candidate Ranking**: Uses the derived Mastery Graph to filter and sort the next best targets. A high frequency of "hint_used" events in the log for a prerequisite node will signal the ranking engine to keep dependent nodes locked or prioritized for reinforcement.
- **Antigravity Planner**: The planner builds study sessions. It consumes candidate lists, which rely on the state derived from the log.
- **Future Reading / Dialogue / Exercise / Assessment Authorities**: These content authorities are event producers. They must conform to the event log schema when emitting records of learner interactions.

## 4. Core Principle

The core architecture of the ULGA learner model relies on strict separation of concerns, moving from immutable raw observations to dynamic, context-dependent planner actions.

```text
[ Raw Interactions ]  ===>  [ Derived State ]  ===>  [ Recommendation ]  ===>  [ Orchestration ]
  Learner Event Log           Learner State            Candidate Ranking         Antigravity Planner
  (Immutable Evidence)        (Aggregated Score)       (Downstream Filter)       (Decision Layer)
```

| Dimension | Learner Event Log | Learner State | Candidate Ranking | Antigravity Planner |
|---|---|---|---|---|
| **Mutability** | **Strictly Immutable**. Write-once, read-many. | **Derived/Rebuildable**. Can be cleared and regenerated from the log. | **Ephemeral**. Computed dynamically at query-time. | **Dynamic / Session-specific**. Adapts to context. |
| **Data Scope** | Single event context (a specific answer, a specific hint, a single timestamp). | Aggregated node-level stats (score, last seen, decay, band). | Sorted list of node candidates filtered by eligibility. | Concrete lesson configuration and pacing. |
| **Logic Content** | No logic. Pure representation of historical fact. | Aggregation and decay logic (Reducers). | Filtering, scoring, and ranking policies. | Constraint checking (gates) and session assembly. |
| **Key Constraint** | Must not alter past events. | Must be fully deterministic based on inputs. | Must not modify the underlying learner state. | Must remain advisory and respect authority gates. |

## 5. Event Taxonomy

To capture the richness of learner behavior without losing granularity, the system defines nine core event types.

```text
                             +------------------------+
                             |   Interaction Events   |
                             +-----------+------------+
                                         |
            +----------------------------+----------------------------+
            |                            |                            |
+-----------v-----------+    +-----------v-----------+    +-----------v-----------+
|    Exposure Events    |    |    Practice Events    |    |   Assessment Events   |
+-----------------------+    +-----------------------+    +-----------------------+
| - exposure_seen       |    | - practice_started    |    | - assessment_attempt  |
| - content_completed   |    | - practice_completed  |    | - mastery_check       |
|                       |    | - answer_submitted    |    |                       |
|                       |    | - hint_used           |    |                       |
|                       |    | - retry_attempt       |    |                       |
+-----------------------+    +-----------------------+    +-----------------------+
```

### 1. `exposure_seen`
- **Definition**: The learner encountered a target node passively or as background context (e.g., reading a passage containing target vocabulary, hearing an introductory explanation).
- **Trigger**: Rendered on page/card load where the target is present but no active production or selection is required.
- **Downstream Effect**: Updates `last_seen_at` and increments exposure counts. Does not count as active practice.

### 2. `practice_started`
- **Definition**: The learner initiated an active practice task focused on one or more target nodes (e.g., clicked "Start Vocabulary Quiz" or loaded a fill-in-the-blank card).
- **Trigger**: Explicit user action starting an exercise.
- **Downstream Effect**: Used to track engagement, drop-out rates, and session boundaries.

### 3. `practice_completed`
- **Definition**: The learner completed an active practice task.
- **Trigger**: Reaching the final card or screen of a practice session.
- **Downstream Effect**: Triggers recalculation of average session performance and marks session-level metadata as complete.

### 4. `answer_submitted`
- **Definition**: The learner submitted a specific answer to an interactive prompt.
- **Trigger**: Clicking "Submit", selecting a multiple-choice option, or submitting speech input.
- **Downstream Effect**: The primary driver of practice-level mastery updates. Evaluates response correctness, calculates response time, and adjusts node-level weights.

### 5. `assessment_attempt`
- **Definition**: The learner completed a high-stakes question or section under test conditions (no feedback, no hints).
- **Trigger**: Submission of an answer inside an assessment module.
- **Downstream Effect**: Updates high-confidence assessment metrics. Bypasses soft practice updates and updates formal CEFR level eligibility.

### 6. `hint_used`
- **Definition**: The learner requested assistance or viewed a hint during a task.
- **Trigger**: Clicking the "Show Hint" button.
- **Downstream Effect**: Acts as a negative or capping modifier on the current event score. Limits maximum achievable mastery for that specific attempt.

### 7. `retry_attempt`
- **Definition**: The learner attempted a task again after a previous failure within the same session.
- **Trigger**: Submission of a response on a card marked as "retry".
- **Downstream Effect**: Tracked separately from first-try success to avoid inflating mastery. Used to measure speed of correction.

### 8. `mastery_check`
- **Definition**: A scheduled, isolated query designed to verify retention of a previously mastered node.
- **Trigger**: System-initiated spacer task presented during a normal session.
- **Downstream Effect**: Directly impacts decay multipliers. A failure here triggers an immediate transition of the node back to the review queue.

### 9. `content_completed`
- **Definition**: The learner finished a complete unit of learning (e.g., finished an entire theme story, read a whole chapter, or finished a dialogue tree).
- **Trigger**: Navigation to the final node of a content bundle.
- **Downstream Effect**: Emits a completion signal for macro-level tracking. Does not directly prove node-level mastery but satisfies progression gates.

Below is the proposed JSON contract structure for the Learner Event Log. It groups related fields logically to ensure validation can be cleanly modularized.

## 7. Evidence Mapping Rules

When raw events are written to the log, they are eventually read by downstream reducers to populate specific evidence containers. Below is the mapping logic used to direct these reductions.

```text
+----------------------+         +------------------------------+
|   Raw Event Type     |         |   Evidence Container Target  |
+-----------+----------+         +--------------+---------------+
            |                                   |
            +-----------------------------------> [Exposure Evidence]
            | (All events)                      | (Tracks last seen & count)
            |                                   |
            +---> [practice_*] -----------------> [Practice Evidence]
            |                                   | (Tracks success rate & time)
            |                                   |
            +---> [assessment_*] ---------------> [Assessment Evidence]
            |                                   | (High confidence levels)
            |                                   |
            +---> [answer_submitted w/ score] --> [Mastery Update Evidence]
            |                                   | (Recalculates decay score)
            |                                   |
            +---> [retry_attempt / hint_used] --> [Reinforcement Evidence]
                                                  (Identifies weak spots)
```

### Exposure Evidence
- **Source Events**: All event types.
- **Rule**: Any log record with `counts_as_exposure: true` maps directly here. 
- **Effect**: Updates the target node's `last_seen_at` field and increments the `exposure_count` inside learner state.

### Practice Evidence
- **Source Events**: `practice_started`, `practice_completed`, `answer_submitted`, `hint_used`, `retry_attempt`.
- **Rule**: Filter events where `counts_as_practice: true` and `attempt_number: 1`. 
- **Effect**: Records attempts, response times, and success ratios within the practicing band. Allows tracking of user engagement trends.

### Assessment Evidence
- **Source Events**: `assessment_attempt`, `mastery_check`.
- **Rule**: Filter events where `counts_as_assessment: true` and `quality_flags.requires_review: false`.
- **Effect**: Ingested with higher weight and confidence score. These updates are resilient to decay because they reflect verified performance under diagnostic constraints.

### Mastery Update Evidence
- **Source Events**: `answer_submitted` (with valid score), `mastery_check`, `assessment_attempt`.
- **Rule**: Filter events where `counts_as_mastery_update: true` and `quality_flags.has_target_nodes: true`.
- **Effect**: Triggers the Learner State Authority to compute a new `mastery_score` using a moving window or Bayesian updater.

### Reinforcement Evidence
- **Source Events**: `hint_used`, `retry_attempt`, or `answer_submitted` (where `is_correct: false`).
- **Rule**: Filter events where `counts_as_reinforcement: true` or where `is_correct: false` / `used_hint: true` occurs.
- **Effect**: Feeds directly into S10J Candidate Expansion. Signals that the learner requires repetition of this opportunity to stabilize the node.

## 8. Allowed / Blocked Downstream Use

To prevent "state leakage" and inflation of mastery scores, downstream consumers must respect strict limits on how different event data is allowed to modify the learner's profile.

### Policy Rules Matrix

| Event Type | Allowed Downstream Action | Blocked Downstream Action |
|---|---|---|
| `exposure_seen` | - Update `last_seen_at`<br>- Increment `exposure_count` | - **Must not** increase mastery score<br>- **Must not** unlock downstream dependencies |
| `hint_used` | - Track as soft/negative evidence<br>- Cap event score at `0.5` | - **Must not** count as full success<br>- **Must not** reset the decay timer |
| `answer_submitted` (score > 0.7) | - Trigger mastery update<br>- Reset decay timer<br>- Count as successful practice | - **Must not** update state if `quality_flags.requires_review` is true |
| `content_completed` | - Mark topic/theme as completed<br>- Satisfy macro progress gates | - **Must not** imply mastery of sub-nodes within the content |
| `retry_attempt` | - Track latency improvement<br>- Record error correction speed | - **Must not** override the initial failure penalty in core mastery |
| `practice_started` | - Log session activity<br>- Track engagement dropouts | - **Must not** update mastery scores or correct counters |

## 9. Data Quality and Guardrails

To prevent corrupt data from rendering learner state unusable, the event log implementation must enforce these strict validator checks:

1. **Event ID Uniqueness**: Every event must have a unique identifier. Duplicate events (detected by comparing `event_id` or `processing_idempotency_key` against a historical index) must be dropped or flagged to prevent score inflation.
2. **Occurred At Parseability**: `occurred_at` must parse as a valid ISO-8601 UTC timestamp. Events with invalid timestamps must be quarantined.
3. **Known Event Type**: The `event_type` must exist in the canonical 9-event taxonomy.
4. **Known Learner ID Format**: The `learner_id` must follow the format `learner:<string_identifier>` (e.g., `learner:usr_00912`). It must not be null or blank.
5. **Known Source Type**: The `source_type` must be an approved value (e.g., `exercise`, `reading`, `dialogue`, `assessment`, `manual`).
6. **Target Nodes Integrity Constraint**: If `evidence_flags.counts_as_mastery_update` is `true`, `target_nodes` must contain at least one valid target identifier inside `vocabulary`, `grammar`, `pattern`, or `chunk`.
7. **Score Validation Constraint**: If `evidence_flags.counts_as_assessment` is `true`, `attempt.score` must be a numeric value between `0.0` and `1.0`. It cannot be null.
8. **No Mutation inside Event Log**: The Event Log writer is strictly prohibited from executing any state mutation logic, score reduction, or database queries against the learner state. Its sole job is to validate schema conformity and append the record.

The introduction of the Learner Event Log will progress through five structured phases:

```mermaid
gridLayout
[ "Phase 1: Schema" ] --> [ "Phase 2: Validator" ] --> [ "Phase 3: Reducer" ] --> [ "Phase 4: Replay" ] --> [ "Phase 5: Ranking" ]
```

### Phase 1: S9Z3 LearnerEventLog Schema Implementation
- **Goal**: Implement the formal JSON schema contract (`ulga/learner_state/learner_event_log_schema.json`) based on the S9Z2 design scan.
- **Deliverable**: Schema file and sample compliant event fixtures.

### Phase 2: S9Z4 LearnerEventLog Validator
- **Goal**: Create the Python utility (`ulga/validators/validate_learner_event_log.py`) to enforce guardrails, check ID uniqueness, and validate ISO-8601 timestamps.
- **Deliverable**: Validator code and a comprehensive unit test suite (`tests/ulga/test_validate_learner_event_log.py`).

### Phase 3: S9Z5 LearnerEventReducer DesignScan
- **Goal**: Outline the reducer architecture. Specify how raw events are processed in chronological order to generate node-level `mastery_score` and `decay_adjusted_score`.
- **Deliverable**: `docs/ulga/ULGA_S9Z5_LEARNER_EVENT_REDUCER_DESIGN_SCAN.md`.

### Phase 4: S9Z6 LearnerState Replay Prototype
- **Goal**: Write the reducer builder (`ulga/builders/build_learner_state_from_log.py`) that reads a log file, executes the reduction logic, and outputs the canonical `learner_state.json` file.
- **Deliverable**: Prototype reducer builder showing correct reconstruction of learner state from simulated event sequences.

### Phase 5: S10A Candidate Ranking Integration
- **Goal**: Connect the Candidate Ranking Engine to the newly generated, event-derived learner state.
- **Deliverable**: Integration tests showing that candidate sorting changes dynamically as simulated practice and assessment events are appended to the log.

### Key Design Decisions
1. **Log Immutability**: Appended events are forever frozen. Any update to scoring formulas requires a full replay of the event log rather than in-place edits to `learner_state.json`.
2. **Schema-Level Routing Flags**: By providing `evidence_flags` directly in the raw event payload, we decouple the log schema from downstream reduction details. The logger simply routes events based on flags set by the content generators.
3. **No Hard-coded Learner Identities**: The contract examples use anonymized format identifiers (`learner:usr_98124b`) to avoid polluting core schemas with specific test user names.
